    Generics #

    Terminology. The term generics usually refers to generic methods and generic types.

    Type variable #

    A type variable is a variable ranging over types.
    For instance, in the chapter dedicated to streams, we repeatedly used the variable $T$ to refer to the type of the elements of a stream.

    Generic method #

    Example #

    Consider the following interface Unit and its two implementations Butterfly and Unicorn.

    Let us assume a method healthiestButterfly with type

    $\qquad$ Butterfly[] $\to$ Butterfly

    that returns one of the healthiest butterflies in the input array (or null if the array is empty).

    The following program does not compile, as expected :

    Butterfly[] butterflies = new Butterfly[]{ new Butterfly(12), new Butterfly(5) };
Unicorn u = healthiestButterfly(butterflies);

    Neither does the following program (as expected still):

    Butterfly[] butterflies = new Butterfly[]{ new Butterfly(12), new Unicorn(5) };
Butterfly b = healthiestButterfly(butterflies);

    Now let us assume that we need to perform the same operation on arrays of unicorns.

    We could write a nearly identical method healthiestUnicorn, with type:

    $\qquad$ Unicorn[] $\to$ Unicorn

    which returns one of the healthiest unicorns in the input array (if any).

    However, this approach is not scalable for a larger number $n$ of units types, because:

    • modifying the implementation requires modifying $n$ methods, and
    • duplicated code means duplicated bugs.

    Alternatively, we could factorize our code by writing a single method healthiest with type:

    $\qquad$ Unit[] $\to$ Unit

    However, with this solution, the two compile-time type checks that we saw above would be lost. The compiler cannot enforce anymore that:

    1. the returned Unit has the same type as the units of the input array,
    2. the input units must have the same type. For instance, the following program would compile:
    Unit[] units = new Unit[]{ new Butterfly(12), new Unicorn(5) };
Unit healthiest = healthiest(units);

    Solution #

    We can factorize our code without sacrificing type safety, by using a type variable $T$ that stands for “any subtype of Unit”. In order words, we can create a method healthiest with type:

    $\qquad T[] \to T$

    where $T$ can be any subtype of Unit.

    Terminology #

    A method with (one or several) type variables is called a generic method.

    Syntax in Java #

    Declaration #

    The generic method in our example can be written as follows:

    <T extends Unit> T healthiest (T[] units) {
    ...
}

    The block <T extends Unit>:

    • declares the type variable T, and
    • specifies that the type Unit is an upper bound on the value of T.

    This variable T can be used not only in the method’s signature, but also in its body.

    Example. We could implement our method in the following (unnecessarily verbose) way. Observe that the type T also appears in the body of the method:

    <T extends Unit> T healthiest (T[] units) {

    T selected = Arrays.stream(units)
                    .max((u1, u2) -> u1.health - u2.health)
                    .orElse(null);
    return selected;
}

    Warning. In this example, T is a (type) variable. So any other variable name could be used, for instance

    <X extends Unit> X healthiest (X[] units) {
   ...
}

    or

    <TypeOfUnit extends Unit> TypeOfUnit healthiest (TypeOfUnit[] units) {
   ...
}

    By convention, the letters T, E and V are often used for type variables (as well as R if the type variable is used for the return type of a method).

    If there is no upper bound on the type variable (in other words, if the upper bound is Object), then the extends statement can be omitted.

    Example.

    <T> T myMethod (T arg1, String arg2) {
   ...
}

    It is also possible to declare several type variables.

    Example.

    <T extends Unit, V> Unit myMethod (T arg1, V arg2) {

}

    Method call #

    A generic method can be called by replacing types variables with concrete types (in a way that is consistent with the method declaration).

    Examples. Let us recall the signature of our generic method above:

    <T extends Unit> T healthiest(T[] units) {
    ...
}

    The following program will compile (and run as expected):

    Butterfly[] butterflies = getButterflies();
Butterfly b = healthiest(butterflies);
Unit u = healthiest(butterflies);

    However, the following program will not compile, as expected :

    Butterfly[] butterflies = getButterflies();
Unicorn u = healthiest(butterflies);

    The following program will not compile either, as expected :

    String[] strings = getStrings();
String u = healthiest(strings);

    Generic type #

    Terminology. A generic type is a type with type variable(s).

    Examples. Let T, E and R be type variables. Then in Java:

    • Set<E> is a generic type,

    • Set<T> is equivalent to Set<E>,

    • Function<T,R> is another generic type.

    Terminology. A parameterized type is a generic type whose type variables are instantiated.

    Examples. In Java:

    • Set<String> is a parameterized type,

    • Function<Unit, Boolean> is another parameterized type.

    Usage #

    Generic types serve the same purpose as generic methods, i.e. factorizing code without sacrificing type safety.

    For some object-oriented languages (like Java or C++), generic types are instrumental in factorizing the implementation of abstract data types (such as lists, sets, associative arrays or queues).

    Example. The Java class HashSet<E> provides a unique instance method to add an element to the set. This method has signature boolean add(E e).

    Syntax in Java #

    Declaration #

    Type variables can be associated to a Java class (resp. interface, record).

    The declaration of these type variables is identical to the one we saw above for methods, but the “diamond” is immediately next to the class (resp. record, interface) name (rather than left of the method declaration):

    Example. We can create a generic wrapper class Box (similar to Java’s Optional), as follows:

    public class Box<T> {
    ...
}

    These type variables are visible in the whole the class (resp. record, interface) body.

    Example.

    public class Box<T> {

    T boxedValue;

    public Box(T boxedValue){
        this.boxedValue = boxedValue;
    }

    T get(){
        if(boxedValue != null) {
            return boxedValue;
        }
        ...
    }
    ...
}

    Warning. The value of the type variables (e.g. the value of T in the above example) may vary from one instance of the generic class to the other. Therefore a type variable cannot be used in a static method (or as the type of a static attribute).

    For instance, the following class does not compile:

    public class Box<T> {

    static T myAttribute;
    ...
}

    Instantiation #

    An instance of a generic class (resp. record) can be created by providing parameter types when calling its constructor.

    Example.

    Box<String> myBoxedString = new Box<String>("banana");

    Syntactic sugar. In this example, the second type parameter String (used with the constructor) is redundant. So it can be safely be omitted:

    Box<String> myBoxedString = new Box<>("banana");

    Instance method call #

    A method declared in a generic class (or interface or record) can be called like a generic method (with the same restriction as the ones described above).

    Example. Let us recall our generic class Box<T>:

    public class Box<T> {

    T boxedValue;

    public Box(T boxedValue){
        this.boxedValue = boxedValue;
    }

    T get(){
        if(boxedValue != null) {
            return boxedValue;
        }
        ...
    }
    ...
}

    The following program will compile (and run as expected).

    Box<String> myBoxedString = new Box<>("banana");
String unboxedString = myBoxedString.get();

    The following program will not compile, as expected :

    Box<String> myBoxedString = new Box<>("banana");
Integer unboxedInteger = myBoxedString.get();

    More examples #

    Consider the following generic class.

    public class Pair<T, V extends T>{
    T arg1;
    V arg2;

    public Pair(T arg1, V arg2){
        this.arg1 = arg1;
        this.arg2 = arg2;
    }

    V someMethod(V mArg){
        return mArg;
    }
}

    Which of the following Java programs compile?

    Pair<Number, Integer> myPair = new Pair<>(15, 32);
Double someNumber = myPair.someMethod(6.0);

    Pair<Number, Integer> myPair = new Pair<>(15, 32);
Number someNumber = myPair.someMethod(6.0);

    Pair<Number, Integer> myPair = new Pair<>(15, 32);
Number someNumber = myPair.someMethod(6);

    Pair<Number, Number> myPair = new Pair<>(15, 32);
Number someNumber = myPair.someMethod(6.0);

    Pair<Integer, Number> myPair = new Pair<>(32, 15);
Integer someNumber = myPair.someMethod(6.0);

    Pair<Integer, Number> myPair = new Pair<>(32, 15);
Number someNumber = myPair.someMethod(6.0);

    Pair<Integer, Integer> myPair = new Pair<>(32, 15);
Integer someNumber = myPair.someMethod(6.0);

    Pair<Integer, Integer> myPair = new Pair<>(32, 15);
Integer someNumber = myPair.someMethod(6);

    Pair<Number, Number> myPair = new Pair<>(32, 15);
Double someNumber = myPair.someMethod(6.0);
