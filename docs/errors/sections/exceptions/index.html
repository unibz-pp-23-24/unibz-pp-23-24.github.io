    Exceptions #

    Programming languages may significantly differ in the way they support runtime errors.

    Java relies on mechanism called exceptions. Several other languages use exceptions, for instance C++, C#, PHP, Python or Ruby.

    An exception is an object or data structure that stores information about an exceptional condition encountered during the execution of a program. In Java, an exception is an object.

    An exception is thrown by a method $m$ when such a condition occurs. The method that called $m$ can either

    • catch the exception (and handle it), or
    • rethrow it.

    Runtime exceptions in Java #

    A runtime exception in Java is an instance of the native class RuntimeException.

    Error stack #

    By default, when a runtime exception is thrown, the program interrupts its execution, and the current call stack is printed on an output stream called System.error.

    For instance, consider the following class:

    public class MyClass{

    void myMethod(City[] city){
        Integer.parseInt("clearlyNotANumber");
    }
}

    When myMethod is executed, it calls the method Integer.parseInt(String string). This method in turn calls the method Integer.parseInt(String string, int radix) with the same string as argument, and a value of 10 for the radix (meaning that the string should represent an integer in base 10). Because the string “clearlyNotANumber” does not represent a number, the method Integer.parseInt(String string, int radix) (which is on top the call stack) throws an exception.

    As a result, the following is printed to System.error:

    java.lang.NumberFormatException: For input string: "clearlyNotANumber"

at java.base/java.lang.NumberFormatException.forInputString(NumberFormatException.java:67)
at java.base/java.lang.Integer.parseInt(Integer.java:668)
at java.base/java.lang.Integer.parseInt(Integer.java:786)
at Myclass.myMethod(Myclass.java:377)
<bottom of the stack>

    Hint. Your IDE allows you to freely navigate through Java’s source code to trace the cause of the exception.

    Hint. To debug your code, by default:

    • find the last method in the error stack that was written by you (in this example Myclass.myMethod)
    • using our IDE, create a breakpoint at the indicated line for this class (XXX in this example)
    • run your program in debug mode (e.g. clicking on the “bug” icon of your IDE)

    Catching an exception #

    If a method $m_1$ throws an exception, then a method $m_2$ in the call stack may catch it.

    I this happens, then the program is not interrupted. Instead, control is passed back to $m_2$.

    More precisely, when an exception is thrown, the JVM “pops” methods calls from the call stack, until either:

    • a catch instruction is found, or
    • the call stack is empty (in which case the program is interrupted).
