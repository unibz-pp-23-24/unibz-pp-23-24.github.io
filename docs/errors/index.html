    Error handling #

    We all make mistakes when writing code.

    Developing efficient strategies to identify mistakes is an essential part of programming.

    Compile time vs runtime errors #

    In (pre)compiled languages (like Java, C/C++, C#, Python, Rust, Go, etc), a distinction is often made between compile time errors and runtime errors

    • a compile time error prevent a program from compiling. It can be thought of as a syntactic error.
    • a runtime error (a.k.a. bug) occurs during an execution of the program. It may depend on the input of the program.

    Hint. Compile time errors tend to be (much) easier to fix than bugs, especially in large projects. Notably, an IDE can identify some compile time errors (and highlight them) before the program is even compiled. So it is often a good practice to write your code in such a way that a maximal number of errors can be identified at compile time.

    Note. For interpreted languages (like Javascript, Lua, etc.), the term “compile time error” is meaningless. However, a distinction can still be made between errors that can be detected at runtime only, and errors that can be preemptively identified (e.g. highlighted by an IDE).

    Error messages #

    A precise error message can significantly simplify debugging.

    As a toy example, consider the following Java program:

    public class City {

  String name;
  int zipCode;

  public City(String name, int zipCode) {
    this.name = name;
    this.zipCode = zipCode;
  }

    public class MyClass{

    void myMethod(City[] city){
        City firstCity = city[0];
        if(firstCity.zipCode > 20000) {
            System.out.println(firstCity.name.length());
        }
    }
}

    Exercise. In this program, there are several reasons why the methodMyClass.myMethod may fail at runtime. Can you identify them?

    Solution

    • the input array may be null,
    • the input array may be empty,
    • the first city in this array may have a zip code $\ge$ 20 000 and no name.

    Example(continued).

    In the above example, if we execute the following:

        myMethod(null);

    then we obtain the following error message:

    java.lang.NullPointerException: Cannot load from object array because "city" is null
	at MyClass.myMethod(MyClass.java:384)

    Note that this message provides a clear indication of:

    • the file (MyClass.java), method in this file MyClass.myMethod and line of code (384) that caused the error,
    • the type of error (NullPointerException),
    • the cause of the error (the variable city has value null).

    Example(continued).

    Similarly, if the method is called with an empty array:

        myMethod(new City[]{});

    We get the following error message:

    java.lang.ArrayIndexOutOfBoundsException: Index 0 out of bounds for length 0
	at MyClass.myMethod(MyClass.java:384)

    Exercise. Which error message will the following call to mymethod produce?

        myMethod(new City[]{new City(30100, null)});

    Solution.

    java.lang.NullPointerException: Cannot invoke "String.length()" because "firstCity.name" is null
	at MyClass.myMethod(MyClass.java:386)

    Invalid inputs #

    When you design a method, its signature may not reflect the valid inputs for the [problem] that it addresses, because:

    • no type exists in the programming language that captures valid inputs, or
    • the method is not a pure function, in the sense that its execution depends not only on its argument, but also on the current state (in memory) of some values, objects, data structures, etc.

    It may be useful to write your own error message for these cases. This may benefit:

    • users of the program, and
    • collaborators (including your future self!) who will maintain (debug, extend, etc.) the program.

    Example. For instance, consider a method

    boolean isSolvable (int[][] grid);

    that takes as input a grid of sudoku, and returns true iff it is solvable, meaning that this grids admits a unique solution. A valid input here may be a 2-dimensional array of integers between 0 and 9, where 0 indicate the absence of value.

    The method could produce an (informative) error message if the input array:

    • is not 9 x 9, or
    • contains a value < 0 or > 9.

    Controlled input #

    Anticipating all invalid inputs may require a lot of boilerplate code. If you have control over the input a method, then this may not be necessary.

    In particular, this may be the case for some auxiliary method (i.e. which are not visible to other classes or components).

    Example (continued). Consider the method boolean isSolvable (int[][] grid) described above. Let us assume that it is private, and called only by a method generateGrid that generates a grid pseudo-randomly. It may be safe to assume that the implementation of generateGrid only produces 9 x 9 arrays with values between 0 and 9.
