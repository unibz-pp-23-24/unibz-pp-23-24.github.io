    Computational cost #

    In this section, we briefly introduce the standard way to estimate the cost of an algorithm, as a function $f(n)$ of the size $n$ of its input.

    Size #

    The size $n$ of the input may be the size of its representation in bits.

    Other common measures are:

    • the number of entries for array,
    • the number of nodes for tree,
    • etc.

    Time or memory #

    Among others, the function $f(n)$ may estimate:

    • the running time of the algorithm, or
    • its memory consumption.

    Worst case, average #

    Among others, the function $f(n)$ may estimate:

    • cost in the worst case, i.e. for the worst possible inputs of size $n$
    • average cost over all possible inputs of size $n$.

    Consider the following algorithm, which returns true iff the input array A contains the input value a.

    boolean contains(int[] A, int a){
    foreach (v in A) {
        if(v == a){
            return true
        }
    }
    return false
}

    Consider all possible inputs where the array has size $n$ (for a given $n$). Among these inputs, what are the worst possible ones?

    How many times would the loop be executed for such an input?

    Consider the following algorithm (known as binary search), which solves the same problem as the previous one, but assuming that the input array is sorted.

    boolean contains(int[] A, input a){
    int begin = 0
    int end = A.lenghth - 1
    while(begin <= end){
        mid = (begin + end) / 2
        v = A[mid]
        if(v <= a){
            end = mid
        } else {
            begin = mid
        }
    }
    return A[begin] == a
}

    Again, for a given array length $n$, what are the worst possible inputs? And how many times would the loop be executed for such an input?

    Asymptotic cost #

    When comparing the costs of algorithms, we are usually interested in scalability, i.e. how they behave when the input becomes large.

    let $f(n)$ measure the (worst case or average) running time of an algorithm.

    If $n$ doubles, does $f(n)$ double as well? I.e. does $f(2 n) = 2 f(n)$?

    If this is the case, then $f$ is a linear function.

    Examples. $f(n) = n$, $g(n) = 5n$ and $h(n) = \frac{1}{2} n$ are linear functions.

    Alternatively, if $f(2 n) > 2 f(n)$, then the cost grows more than linearly with the size of the input.

    Examples.

    • if $f(n) = n^2$, then $f(2n) = 4 f(n)$,
    • if $f(n) = c n^2$ for some c, then $f(2n) = 4 f(n)$ as well,
    • if $f(n) = n^3$, then $f(2n) = 8 f(n)$,
    • if $f(n) = 2^n$, then $f(n+1) = 2 f(n)$
    Alternatively, $f(n)$ may grow less than linearly with $n$. For instance:

    Examples.

    • if $f(n) = \sqrt{n}$, then $f(4n)$ = 2f(n)$
    • if $f(n) = \sqrt[3]{n}$, then $f(8n)$ = 2f(n)$
    • if $f(n) = \log_2 n$, then $f(n+1) = 2 f(n)$

    Comparison of cost functions #

    Two cost functions $f$ and $g$ have the same asymptotic behavior if they intuitively have the same growth.

    Examples. All linear functions have the same asymptotic behavior.

    Similarly (as illustrated above), the functions $f(n) = n^2$ and $g(n) = 3n^2$ have the same asymptotic behavior.

    Terminology. A function $f: X \to Y$ is monotone if $x_1 \le x_2$ implies $f(x_1) \le f(x_2)$ for any $x_1, x_2 \in X$.

    Definition. Let $f$ and $g$ be two monotone functions over $\mathbb{n}$ such that $\lim_{n \to \infty} \frac{f(n)}{g(n)}$ is defined. And let $l$ be this limit.

    • if $l = 0$ then $g$ outgrows $f$,
    • if $l = \infty$ then $f$ outgrows $g$,
    • otherwise (i.e. if $l \in \mathbb{R}$ and $l \neq 0$) $f$ and $g$ are asymptotically equivalent.
    Hint. $f$ outgrows $g$ iff the derivative of $f$ outgrows the derivative of $g$.

    Example. let $f(n) = 5n^2$ and let $g(n) = n^3$.

    Then $\lim_{n \to \infty} \frac{f(n)}{g(n)} = \lim_{n \to \infty} \frac{5n^2}{n^3} = \lim_{n \to \infty} \frac{10n}{3n^2} = \lim_{n \to \infty} \frac{10}{6n} = 0$

    So $g$ outgrows $f$.

    Example. let $f(n) = 5n^2$ and let $g(n) = n^2$.

    Then $\lim_{n \to \infty} \frac{f(n)}{g(n)} = \lim_{n \to \infty} \frac{5n^2}{n^2} = \lim_{n \to \infty} \frac{10n}{2n} = 5$

    Since $5 > 0$, we can conclude that $f$ and $g$ are asymptotically equivalent.

    Hint. If there exists a function $h$ over $\mathbb{N}$ such that

    • $\lim_{n \to \infty} h(n) = \infty$ and
    • $f(n) = g(n)h(n)$,

    then $f$ outgrows $g$.

    Example. let $f(n) = n \cdot log_2 n$ and let $g(n) = n$.

    Take $h(n) = log_2 n$. Then

    • $\lim_{n \to \infty} h(n) = \lim_{n \to \infty} \log_2(n) = \infty$, and
    • $f(n) = n \cdot log_2 n = g(n) h(n)$,

    Therefore $f$ outgrows $g$.

    Equivalence classes #

    Notation. If $f$ is a cost function, then $\Theta(f)$ is used to denote the set of all cost functions that are asymptotically equivalent to $f$ (including $f$ itself).

    Example. If $f(n) = n$, then the class $\Theta(f)$ contains the functions:

    • $g(n) = 2n$,
    • $h(n) = \frac{n}{2}$,
    • $j(n) = 3n + 4$,
    • etc.
    Note. If $g$ is in $\Theta(f)$, then $\Theta(f)$ and $\Theta(g)$ are the same class.
    Notation. If $\Theta(f)$ is a class of cost functions, then the name of the function ("$f$" here) is often omitted to refer it. The definition of $f$ is often used instead, with $n$ as variable.

    Examples.

    • if $f(n) = n$, then the class $Theta(f)$ is usually dented with $Theta(n)$,
    • if $f(n) = n^2$, then the class $Theta(f)$ is usually denoted with $Theta(n^2)$,
    • etc.

    A simple, representative function in each class is usually preferred to denote it.

    Example. $Theta(n)$, $\Theta(\frac{n}{2})$ and $Theta(3n+4)$ are three alternative notations for the same class of cost functions.

    For readability, the former is usually preferred.

    Some of the most frequent classes of algorithm cost functions are (from lower to higher cost):

    • $\Theta(1)$ (a.k.a. constant cost)
    • $\Theta(\log n)$ (a.k.a. logarithmic cost)
    • $\Theta(n)$ (a.k.a. linear cost)
    • $\Theta(n \log n)$
    • $\Theta(n^2)$ (a.k.a. quadratic cost)
    • $\Theta(n^3)$ (a.k.a. cubic cost)
    • $\Theta(2^n)$

    Useful simplifications #

    The definition above imply that the cost function of an algorithm can often be simplified into an equivalent one, using simple rules.

    Hint. If $f(n) = c \cdot g(n)$ for some constant $c$, then $f(n)$ is in $\Theta(g)$.

    Examples.

    • $f(n) = 3n$ is in $\Theta(n)$
    • $f(n) = 2n^3$ is in $\Theta(n^3)$
    • $f(n) = 5 \log n$ is in $\Theta(\log n)$
    • etc.

    Hint.

    • if $h(n) = \log_a n$ (for some base $a$), then $h$ is in $\Theta(\log n)$
    • if $h(n) = n \log_a n$ (for some base $a$), then $h$ is in $\Theta(n \log n)$

    Examples.

    • $f(n) = \log_2 n$ is in $Theta(\log n)$
    • $f(n) = \frac{1}{5} \log_3 n$ is in $Theta(\log_3 n) = $Theta(\log n)$
    • $f(n) = 4n \log_2 n$ is in $Theta(n \log_2 n) = $Theta(n \log n)$

    Hint. If $f(n)$ is a sum of terms (e.g. a polynomial), then it belongs to the class that represents the highest order term.

    Examples.

    • $f(n) = 2n^2 + 3n + 20$ is in $\Theta(2n^2) = \Theta(n^2)$
    • $f(n) = \frac{3}{2}n^3 + 2n^2 + 3n + 20$ is in $\Theta(n^3)$
    • $f(n) = 2 n\log(n) + 3n + \sqrt{n}$ is in $\Theta(n \log n)$

    Big $O$ #

    Notation. $O(f)$ denotes the union of all classes of functions that are at most as expensive as $f$ (asymptotically).

    Example. $f(n) = 3n + 5$ is in $\Theta(n)$, therefore the following also hold:

    • $f$ is in $O(n^2)$
    • $f$ is in $O(n^2 \log n)$
    • $f$ is in $O(n^{2.5})$
    • $f$ is in $O(n^3)$
    • $f$ is in $O(2^n)$

    Note. As a consequence,

    $\qquad f$ is in $\Theta(g)$

    is syntactic sugar for

    $\qquad f$ is in $O(g)$ and $g$ is in $O(f)$

    Overloaded equality #

    Warning. Instead of “$f$ is in $\Theta(g)$”, it is customary to write:

    $\qquad f = \Theta(g)$

    (and similarly for $O$), thus overloading the meaning of the symbol “$=$”.

    Examples.

    • $f(n) = 3n^2 + 5 = \Theta(n^2)$
    • $f(n) = 2 n^2 \log n = \Theta(n^2 \log n)$
    • $f(n) = 2 n^2 = O(n^2)$
    • $f(n) = 2 n^2 = O(n^3)$

    Application #

    Let us now evaluate the asymptotic cost of some algorithms.

    For simplicity, we will focus of the worst-case running time (but the same notion of asymptotic cost function is used for average cost and/or memory consumption).

    RAM cost model #

    $ The so-called Random Access Machine model of computation makes several assumptions about computational cost. In particular, for a given index $i$, the cost of accessing the $i$^th element of an array does not depend on the size of this array.

    Another simplification that is often made (although not consistently by all authors) in the RAM model is that the cost of a value comparison or basic arithmetic operation (e.g. addition or mutiplication) does not depend on the values of its operands.

    Example. Consider the following method

    boolean contains(int[] A, int i){
    return A[i] + A.length;
}

    Let $n$ be the size of the input array $A$, and let us express the running time of this method as a function of $n$. Observe that:

    • the cost $c_1$ of reading the value $A[i]$ is independent of $n$ (in the RAM cost model),
    • the cost $c_2$ of reading the length of A is also independent of $n$
    • the cost $c_3$ of summing these two values is also independent of $n$

    Therefore expressed as a function $f$ of $n$, the cost of this method is

    $\qquad f(n) = c_1 + c_2 + c_3$,

    which is in $Theta(1)$.

    In other words, the running time of this method is constant in the size of the array.

    Example. Consider the same algorithm as above (but where the meaning of for each is made explicit)

    boolean contains(int[] A, int a){
    for (int i = 0; i < A.length; i++) {
        if(A[i] == a){
            return true
        }
    }
    return false
}

    Let $n$ be once again the size of the input array A, and let us express the worst case running time of this method as a function of $n$.

    Recall that the worst case is the one where the input value a does not appear in A. In this case, the loop is executed $n$ times.

    Observe that (in the RAM cost model):

    • the cost $c_1$ of one execution of the loop is independent of $n$, and
    • the cost $c_2$ of the final return statement is also independent of $n$.

    Therefore, expressed as a function $f$ of $n$, the cost of this method is

    $\qquad f(n) = c_1 n + c_2$,

    which is in $Theta(n)$.

    In other words, the worst-case running time of this method is linear in the size of the array.

    Exercise. What is the asymptotic running time of the binary search algorithm seen above, expressed as function of the size of A?

    Solution.

    The running time of this algorithm is logarithmic in the size $n$ of the array (i.e. in $Theta(\log n)$).

    Exercise. What is the asymptotic running time of the following method, expressed as function of the size of A?

    Can you think of a more efficient algorithm for the same problem (and what would be its running time)?

    int myMethod(int[] A){
    int val = 0;
    for (int i = 0; i < A.length; i++){
        for (int j = 0; j <= A.length ; j++){
            val += A[i] + A[j]
    }
    return val;
}

    Solution

    The method has quadratic running time (i.e. its running time is in $Theta(n^2)$).

    A more efficient algorithm (with linear running time ) is:

    int myMethod(int[] A){
    int val = 0;
    for (int i = 0; i < A.length; i++){
        val += A[i]
    }
    return val * val;
}

    Comparing two sorting algorithms #

    Selection sort #

    The following algorithm is called selection sort.

    It sorts the input array A by building an initial segment A[0 .. i] that consists of the sorted i + 1 least elements. Each iteration of the main loop:

    1. selects the least min in the remainder of the array (i.e. in the segment A[i + 1 .. A.length]),
    2. appends min to the sorted segment.

    In the worst case, how many times is the inner loop (i.e. the loop of the auxiliary method selectMin executed)? What does this tell us about the wort case asymptotic running time of this algorithm?

    Mergesort #

    Mergesort is another sorting algorithm, that we discussed when studying recursion.

    We reproduce it here for convenience:

    int[] mergesort(int[] A) {

    // Base case: the input array has size 1, it is already sorted.
    if(A.length == 1){
        return A
    }

    // Inductive case (the input array has size > 1)
    // Compute the middle index (rounded down)
    int m = (A.length - 1) / 2

    // Copy the left half of A into a new array
    Type[] leftHalf = copy(A[0 .. m])
    // Sort it
    leftHalf = mergesort(leftHalf)

    // Copy the right half of A into a new array
    Type[] rightHalf =  copy(A[m + 1 .. A.length - 1])
    // Sort it
    rightHalf = mergesort(rightHalf)

    // merge the two sorted halves
    return merge(leftHalf, rightHalf)
}

    Let us analyze the worst-case asymptotic cost of this algorithm, as a function of the size $n$ of the initial input array. For simplicity, we will assume that $n$ is a power of two.

    First, let us analyze the cost of this method, omitting the two recursive calls.

    1. In the base case, the cost is a constant $c_0$

    2. In the inductive case, let $s$ be the size of the current input array A.

    3. computing the middle index m has constant cost $c_1$,

    4. copying both halves of the input array into two new arrays has linear cost, i.e. $c_2 s$ for some constant $c_2$,

    5. merging the two halves halves has linear cost, i.e. $c_3 s$ for some constant $c_3$.

    Now consider the following tree, where each node represents a recursive calls to mergesort, labeled with the corresponding input size. In this picture $n$ stands for the size of the initial array.

    Observe that in this tree, the cumulated size of the arrays at each depth is $n$. For instance:

    • $n$ at depth 0,
    • $2\frac{n}{2}$ at depth 1,
    • $4\frac{n}{4}$ at depth 2,
    • etc.

    We can decompose the total running time as follows:

    1. Base cases. The base case is execute $n$ times, each with a constant cost $c_0$. So the cumulated cost for the deepest level of the tree is $c_0 n$

    2. Inductive cases.

      1.1. The middle index m is computed once for each non-leaf node, i.e. 1 + 2 + 4 + 8 + .. + \frac{n}{2} = n-1$ times. So the cumulated cost for this operation is $c_1 n-1 = c_1 n - c_1$.

      1.1. The cost of copying the input array into two new arrays is 1.1.1. $c_2 n$ at depth $0$ 1.1.1. $2 c_2 \frac{n}{2} = c_2 n$ at depth $1$ 1.1.1. $4 c_2 \frac{n}{4} = c_2 n$ at depth $2$ 1.1.1. etc. Since the tree has depth $\log_2 n$, the cumulated cost of these operations is $c_2 n \log_2 n$ 1.1. Similarly, the cumulated cost of the merging operations is $c_3 n \log_2 n$

    Summing up all these costs, we get:

    $$ \qquad f(n) = c_0 n + c_1 n - c_1 + c_2 n \log_2 n + c_3 n \log_2 n =

    (c_2 + c_3) n \log_2 n + (c_0 + c_1) n - c_1 $$

    which is in $Theta(n \log n)$.
