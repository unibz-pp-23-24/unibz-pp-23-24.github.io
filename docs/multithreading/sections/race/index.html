    Race condition #

    Definition. A race condition occurs when the outcome of a program may vary depending on the order in which two threads access a resource.

    Illustrations #

    Example. The following program apparently increaments variable x twice. Howver, it may ouput either 1 or 2, dependending on the execution environemnt:

    int i = 0

spawn i++

i++

sync

print(i)

    The reason is that a processor core does not increment the value of a variable as an atomic operation. Instead, this is a sequence of three instructions:

    1. read the value of i and fetch it to one of the core’s registers.
    2. increment the value of this register.
    3. write the register’s value back to memory.

    If a context switch occurs during the execution of these three operations, or if the two thread are executed in parallel (i.e. on different cores), then the outcome may be unexpected. Among other possibilities, the following sequence of actions may be performed:

    • the current executes step 1, copying the value 0 from memory to a register,
    • then the child thread executes step 1 as well, copying the value 0 to another register,
    • then the current thread executes steps 2 and 3, writing the value 1 back to memory.
    • then the child thread executes step 2 and 3, writing the value 1 back again to memory.

    Note. A race condition can only occur if two threads access a same resource and at least one of them modifies it.

    Example. The following algorithm performs a matrix by vector multiplication with a “parallel” loop, and is free of race conditions:

    product(int[][] M, int[]v){
    int[] w = new int[M.length]
    for parallel (int i = 0; i < M.length; i++){
        for (int j = 0; j < v.length; j++){
            w[i] += M[i][j] * v[j]
        }
    }
    return w
}

    However, adding the keyword parallel to the inner loop may yield an incorrect output.

    product(int[][] M, int[]v){
    int[] w = new int[M.length]
    for parallel (int i = 0; i < M.length; i++){
        for parallel (int j = 0; j < v.length; j++){
            w[i] += M[i][j] * v[j]
        }
    }
    return w
}

    The reason is that two threads may now modify the value of w[i] concurrently, for the same i (but different values of j).

    Exercise. Modify this method so that the inner loop is executed in a concurrent fashion, but ruling out race conditions.

    Bugs #

    Warning. In general, a bug caused by a race condition cannot be consistently reproduced, even on the same hardware.

    This is why race conditions are notoriously difficult to identify and/or debug, even for software that has been extensively tested (an example of a race condition with dramatic impact is the Northeast amerixca blackout of 2003).

    Caution #

    Multithreading is a good example (together for instance with caching the results of a computation) of an error-prone optimization technique, which may have an impact on code readability and correctness.

    A common mistake made by less experienced developers is to use multithreading “because we can”.

    Instead, it is usually recommended to use multithreading cautiously, and only if a performance bottleneck has been identified in an application (e.g. with a profiler): unacceptable processing time, non-responsive GUI, etc.

    We reproduce here three citations made in Effective Java (Item 67, “Optimize judiciously”):

    More computing sins are committed in the name of efficiency (without necessarily achieving it) than for any other single reason—including blind stupidity.

    William A. Wulf, 1972

    We should forget about small efficiencies, say about 97% of the time: premature optimization is the root of all evil.

    Donald Knuth, 1974

    We follow two rules in the matter of optimization: Rule 1. Don’t do it. Rule 2 (for experts only). Don’t do it yet — that is, not until you have a perfectly clear and unoptimized solution.

    M. A. Jackson, 1975

    Precedence #

    Let $a$ (resp. $b$) be an instruction executed in a thread $A$ (resp. $B$).

    By default, if $A$ and $B$ are executed concurrently, there is no guarantee that $a$ is executed before $b$. However, we can enforce it.

    in Java #

    The Javadoc of the package java.util.concurrent summarizes the situations that ensures that $a$ is executed before $b$.

    We only present here a few of them.

    Synchronized methods #

    A java method can be declared as synchronized, as follows:

    public synchronized int myMethod() {
    ...
}

    For an instance method $m$, this has two consequences. Let $o$ be and object, let $A$ and $B$ be two threads, and let use assume that $A$ executes $o.m$ (i.e. executes the method $m$ for object $o$). Then:

    • $B$ cannot execute any synchronized method for $o$ before the termination of $o.m$,
    • after completion of $o.m$, if $m$ modified $o$, then these modifications will be visible to any synchronized method executed for $o$ by $B$).

    For a static method, the effect is the same, the object $o$ is the instance of Class that represents the class itself.

    Warning. A constructor cannot be synchronized (the program will not compile).

    SynchronizedRGB color = new SynchronizedRGB(0, 0, 0, “Pitch Black”); … int myColorInt = color.getRGB(); //Statement 1 String myColorName = color.getName(); //Statement 2

    If another thread invokes color.set after Statement 1 but before Statement 2, the value of myColorInt won’t match the value of myColorName. To avoid this outcome, the two statements must be bound together:

    synchronized (color) { int myColorInt = color.getRGB(); String myColorName = color.getName(); }
